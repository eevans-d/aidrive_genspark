- Resultado:
  - T02 ya implementa un mutex DB-backed via PATCH atomico condicional:
    - `PATCH facturas_ingesta SET estado='extrayendo' WHERE id=X AND estado IN ('pendiente','error')`
    - Si PATCH retorna 0 filas → la factura ya esta siendo procesada → 409
  - Este mecanismo opera a nivel de PostgreSQL (motor de BD), garantizando serializacion incluso con multiples instancias de Deno Edge Runtime
  - Un mutex in-memory por isolate Deno NO seria efectivo porque Edge Functions usan isolates efimeros y multiples replicas
  - facturas-ocr ahora valida que el estado sea `extrayendo` (set por gateway), `pendiente` o `error`, lo que previene doble procesamiento
  - El lock se libera al completar (estado→extraida) o al fallar (estado→error)
- Evidencia:
  - `supabase/functions/api-minimarket/index.ts:2343-2358` (PATCH atomico)
  - `supabase/functions/facturas-ocr/helpers.ts:52` (extrayendo en VALID_FACTURA_OCR_EXTRAER_ESTADOS)
  - Tests: `tests/unit/ocr-concurrency-lock.test.ts` (8 PASS, cubriendo race condition)
- Riesgos/rollback:
  - Bajo. El lock es conservador: si el proceso falla, el gateway resetea estado a error
  - Timeout safety: si el edge function muere sin actualizar, el gateway detecta `extrayendo` y lo resetea a `error`
